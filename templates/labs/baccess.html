Insecure Direct Object Reference Explained

Insecure Direct Object Reference (IDOR) is a type of Broken Access Control (BAC) vulnerability that occurs when an application exposes internal object identifiers (such as database IDs) without verifying whether the current user is authorized to access them. Instead of enforcing ownership checks, the application trusts user-supplied references, allowing attackers to access or manipulate resources belonging to other users

Attackers can manipulate object identifiers to:

  • access other users' private data
  • modify records they do not own
  • perform unauthorized actions
  • enumerate sensitive resources
  • escalate privileges indirectly

IDOR is one of the most common real-world access control failures because developers often assume that authentication alone is sufficient, without validating object-level permissions.

Other BAC vulnerabilities include:

  • privilege escalation (user => admin)
  • forced browsing of restricted endpoints
  • missing role checks
  • insecure API object references
  • bypassing multi-tenant boundaries

All BAC vulnerabilities share one core issue: the application fails to properly verify that a user is allowed to perform an action on a specific resource.

Vulnerable Object Reference Example

When retrieving a poll, a developer may implement the following server-side logic:

poll = SandboxPoll.objects.get(id=poll_id)

Because the query only checks the poll's ID, any user who can guess or enumerate IDs may access other users' sandbox records.

For example, changing the URL:

/labs/baccess/polls/detail/2/

to

/labs/baccess/polls/detail/11/

may grant access to objects (polls) owned by other users as has been demonstrated in this lab.

Secure Approach

The secure implementation enforces object-level authorization by ensuring users can only access resources tied to their session. For instance:

poll = SandboxPoll.objects.get( id=poll_id, session_id=request.session["lab_session"])

By validating both the object identifier and the session ownership, the application ensures users can only interact with resources created within their own sandbox environment.

This prevents unauthorized data exposure even when object IDs are predictable or sequential.

Key takeaway: Never trust user-controlled object identifiers alone. Always verify that the requested resource belongs to the current user, session, or authorized role before allowing access.
